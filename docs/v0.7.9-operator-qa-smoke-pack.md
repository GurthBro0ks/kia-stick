# KIA Stick v0.7.9 Fake-Only Operator QA Smoke Pack

Phase: `KIA-Stick-v0.7.9-fake-only-operator-qa-smoke-pack`

Product version: `0.7.0`

Prompt version: `prompt.fake-docs.v0.5-import-wizard-hardening`

## Purpose

This smoke pack makes local operator QA repeatable for the accepted v0.7 fake-only state. It is a docs/tests/tooling phase only. It does not change runtime UX, approve real-doc work, or add any document intake capability.

The checklist is written for local manual or browser review, and the helper script is intentionally narrow: it checks repo contract text plus optional local app routes only.

## Surfaces To Smoke

### Chat

- Chat renders as the first operator surface.
- Smoke label: fake answer/save/no-answer behavior is safe.
- Fake prompts produce citation-first answers with mode, scope, detail, build identity, prompt identity, and provider nearby.
- Saving a citable fake answer creates or refreshes a Saved record with current metadata.
- No-answer responses stay explicit and cannot create a Saved record.

### Sources

- Sources shows fake source IDs, hierarchy ranks, citable/context labels, and source class labels.
- Source review shows prompt version and display build traceability.
- Citable fake sources are distinguishable from context-only fake material.

### Saved

- Empty state explains that no fake answer has been saved yet.
- Saved detail state shows productVersion, promptVersion, display build, provider, mode, scope, detail, and timestamp.
- Duplicate save feedback does not create duplicate records.

### Upload

- Upload remains fake metadata buttons only.
- Operator can confirm fake sample metadata only, then queue a fake sample or fake batch.
- Smoke proof must confirm no chooser opens and no bytes are read.

### Import

- Import remains a fake wizard scaffold.
- Current step, next step, deterministic blocked action, and export safety labels are visible.
- Blocked actions remain deterministic and fake-only.

### Vault

- Vault keeps dense but readable fake metadata rows.
- Workflow state, redaction review label, metadata review label, index gate label, audit-export label, safety label, and next-step label are visible near each record.
- Quarantine, redaction review, and metadata review never imply index approval.

### Settings

- Settings exposes productVersion, displayVersion, promptVersion, provider, corpus, index, and fake-only privacy boundary.
- Settings does not become an approval surface for real-doc work.

### `/health`

- `/health` returns the current phase, target machine, fake-only flags, productVersion, displayVersion, promptVersion, provider, corpusVersion, indexVersion, buildDate, and gitSha.
- `fakeOnly` remains true and `realDbTouched` remains false.

### `/version`

- `/version` shows displayVersion prominently.
- Runtime metadata includes productVersion, channel, buildDate, gitSha, corpusVersion, indexVersion, promptVersion, and provider.

### Mobile / Narrow View

- Bottom nav remains visible and readable.
- Mode/scope/detail chips, repeated cards, safety labels, and proof labels wrap without hiding the fake-only boundary.

## Local Smoke Helper

Run:

```bash
npm run operator:smoke
```

Optional local route check:

```bash
npm run operator:smoke -- --base-url http://127.0.0.1:3000
```

The helper defaults to `http://127.0.0.1:3000`. It only accepts loopback app URLs. If no local app server is reachable, it still verifies the repo-owned smoke contract and reports local route checks as skipped. Use `--require-server` when an operator wants the helper to fail unless the app is already running locally.

## Safety Boundary

This phase does **not** approve real-doc work. It does not add file pickers, directory pickers, drag/drop import zones, path readers, real upload handlers, OCR, embeddings, vector stores, real quarantine, real redaction, real indexing, private-data commits, services, secrets, or private document access.

`queue-015-v07-first-real-doc-gate-request` remains blocked. Any future real-doc path still needs separate approval with exactly one gate, exactly one document, and the existing safety packet/checklist policy.
