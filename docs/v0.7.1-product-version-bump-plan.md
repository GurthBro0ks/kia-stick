## Purpose

KIA Stick still exposes runtime `productVersion` `0.4.0` even though later phases added fake-only import workflow hardening, local proof automation, task queue tooling, release checks, v0.6 real-doc safety planning, and the v0.7 decision checkpoint. This plan decides how a future, separately approved runtime/product milestone bump should be selected and validated.

This document is planning only. It does not start the bump and does not authorize real-document implementation.

## Bump Target Comparison

| Candidate | What it would mean | Strengths | Risks |
| --- | --- | --- | --- |
| `0.5.0` | Conservative catch-up to the fake import wizard and release-readiness era. | Avoids a visible semver skip and maps to the first major post-0.4 user-facing fake-workflow milestone. | Understates the accepted v0.6 safety-planning and v0.7 decision work; may require another bump soon. |
| `0.6.0` | Product identity follows the real-doc planning and safety-governance era. | Acknowledges the size of the v0.6 governance work and the safety artifacts now in the repo. | Can imply real-doc capability exists, which is false; risky unless release notes loudly state planning-only/no implementation. |
| `0.7.0` | Runtime identity catches up to the v0.7 checkpoint after v0.5 and v0.6 work stayed intentionally held at `0.4.0`. | Best matches the current backlog state, keeps the actual later bump to one clean milestone, and avoids overloading `0.6.0` with real-doc implication. | Skips visible runtime milestones `0.5.0` and `0.6.0`; needs a clear release note explaining the hold history. |

## Recommended Decision Rule

Recommended default target: `0.7.0`, but only in a later, separately approved bump implementation phase.

Use this rule:

1. Choose `0.7.0` if the later phase is a pure product identity catch-up release, no new runtime capability is added, and release notes explain that `0.5.x` and `0.6.x` were accepted while `productVersion` was intentionally held at `0.4.0`.
2. Choose `0.5.0` if the operator wants strict chronological semver and the release note focuses only on fake import wizard/readiness work.
3. Choose `0.6.0` only if the release note is explicitly framed as safety/governance planning and repeatedly states that no real-doc implementation exists.
4. Choose no bump if the later phase is docs/tests/state only, validation is not clean, or the version would imply unimplemented real-doc capability.

## Blockers Before Actual Bump

A later bump phase must not proceed until all blockers below are cleared:

- A future prompt explicitly approves changing runtime/product version constants.
- The selected bump target is named exactly: `0.5.0`, `0.6.0`, `0.7.0`, or no bump.
- A GitHub-safe release note explains why prior work did or did not warrant a runtime product bump.
- `lib/version.ts`, `package.json`, `feature_list.json`, `README.md`, and `CLOSEOUT.md` have a single coherent product-version story.
- `/health`, `/version`, visible UI versioning, and saved-answer metadata are validated as one contract.
- `promptVersion` remains unchanged unless a future prompt separately approves a prompt bump.
- No file picker, path reader, file read, copying, OCR, upload handler, real redaction, embeddings, indexing, vector store, private-vault inspection, or real-document access is added.
- All tests, release checks, privacy scans, fake-doc scans, and proof checks pass.
- The worktree has no unrelated dirty files.
- No secrets, private paths, raw document text, snippets, session data, account identifiers, or private evidence are committed.

The later bump implementation phase must run and save proof for every item below:

   - Confirm `feature_list.json` release readiness fields equal the selected target.
   - Confirm `PROMPT_VERSION` remains `prompt.fake-docs.v0.5-import-wizard-hardening` unless a separate prompt approves otherwise.
   - Confirm `/health` reports the selected `productVersion`.
   - Confirm `/version` displays the selected `productVersion`.

3. Required commands:
   - `npm run release:check`

   - `npm run proof:latest`

   - `npm run closeout:summary`
   - `npm run scan:fake`

4. Runtime capability guard checks:
   - No browser file input.
   - No `showOpenFilePicker`.
   - No directory picker.
   - No `FileReader`.
   - No server-side real document reader.
   - No OCR path.

   - Proof contains no raw private source path, raw document text, filename, snippet, OCR text, hash of sensitive content, token, cookie, account identifier, device identifier, session data, upload, export, backup, vector data, embedding data, or private note.

## Rollback And No-Bump Criteria

Do not bump, or roll back the bump before push, if any of these occur:

- `npm run release:check` fails.
- `npm run qa` fails.
- The version surfaces disagree.
- The prompt version changes without separate approval.
- The bump target is ambiguous.
- The bump would imply real-doc implementation.
- Any runtime real-doc capability appears.
- Any private path, secret, raw content, snippet, account/session/device data, or unsafe proof output is present.
- The worktree has unrelated dirty files.
- `git diff --check` fails.
- The operator has not approved the selected target in a later bump implementation prompt.

If a bad bump is discovered after commit but before push, revert that local bump commit and rerun validation. If a bad bump is discovered after push, create a follow-up rollback commit that restores the prior product version, documents the reason, and reruns the full checklist.

## GitHub-Safe Proof Rules

Allowed proof fields:

- Phase name.
- Commit SHA.
- Command names.
- Product and prompt version strings.
- PASS/WARN/FAIL states.
- Gate names and outcome labels.
- Counts by safe category.
- Booleans such as `real_document_access=false`.

Forbidden proof fields:

- Raw document text.
- Snippets.
- Private source paths.
- Filenames when identifying private sources.
- Member, employee, witness, officer, case, account, session, device, contact, or credential identifiers.
- OCR text.
- Hash values for sensitive content.
- Exports.
- Backups.
- Private databases.
- Vector data.
- Embeddings.
- Private notes.
- Screenshots of real content.
- Any field that lets a reviewer reconstruct or locate a private source.
