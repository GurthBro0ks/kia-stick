# KIA Stick v0.6.5 Local-Only Redaction Policy Plan

Phase: `KIA-Stick-v0.6.5-local-redaction-policy-plan`

Status: `PLAN ONLY`

Baseline: `origin/main` verified at `8ae4dd0`, the pushed v0.6.4 operator approval packet.

Product version impact: none. `productVersion` stays `0.4.0`; any future product milestone bump must be approved in a separate implementation phase. This plan only documents that a future bump would be future-prompt-only.

Model recommendation: Claude for the redaction policy and safety wording. GPT/Codex is acceptable for docs and tests. GLM-5.2 is optional for a second-opinion review only.

## Purpose

This document plans the **local-only redaction review policy** for any future one-document real-doc pilot. It defines redaction categories, PASS/WARN/FAIL handling, reviewer roles, escalation rules, default-deny and not-indexable behavior, and deletion/retention rules.

This plan is policy design only. It does not detect, redact, transform, or store any real content. A completed plan informs a future reviewed gate; it does not start one.

## Plan-Only Boundary

This policy plan is a planning artifact only. It does not authorize or implement real-document reads, directory listing, file selection, file pickers, path reading, copying, OCR, text extraction, summarization, transformation, detection over real content, real redaction, embeddings, indexing, vector storage, upload handling, private-vault inspection, exports, service changes, pushes, or private-data commits.

No real document path, filename, text, snippet, image, OCR output, identifier, account data, session data, hash, export, vector data, or private note is needed to complete this phase, and none may be pasted into a completed plan or its proof.

The only approved output of this phase is GitHub-safe docs, tests, queue state, and closeout state.

## Policy Defaults

- **Default deny.** Any content not yet reviewed and cleared is treated as sensitive and blocked.
- **Default not-indexable.** Every document starts `not_indexable`; eligibility can change only after copy, redaction, metadata, and operator gates all pass in a later reviewed phase.
- **Detection is not approval.** Automated detection (in a future phase) only proposes candidates; a human reviewer must confirm every redaction outcome.
- **No private content in proof.** Only labels, counts, booleans, gate names, and PASS/WARN/FAIL states may enter GitHub-safe proof.
- **Local only.** All future review happens locally; no cloud calls, uploads, or external services.

## Redaction Categories

Every future one-document review must classify content against all of the following categories. Each category defaults to **redact unless cleared by a reviewer**.

| Category | Default | Notes |
| --- | --- | --- |
| Member identifiers | redact | Member names, member numbers, union IDs. |
| Employee IDs | redact | Employee/clock/badge numbers and equivalent. |
| Contact info | redact | Phone, email, mailing/home address, emergency contacts. |
| Case facts | redact | Specific facts that identify a grievance, dispute, or investigation. |
| Medical | redact | Health conditions, restrictions, FMLA, medical documentation. |
| Discipline | redact | Disciplinary actions, notices, records, PDIs. |
| Settlement | redact | Settlement terms, amounts, and conditions. |
| Witnesses | redact | Witness names, statements, and identifying details. |
| Screenshots and images | redact | Any embedded image, photo, scan, or screenshot of real content. |
| Management and officer names | redact | Supervisor, manager, and union officer names and titles when identifying. |
| Dates | review | Event/incident/hearing dates that could re-identify a case; redact when identifying. |
| Locations | review | Facility, station, route, and site references; redact when identifying. |
| Installation data | redact | Installation/facility codes, finance numbers, plant identifiers. |
| Grievance IDs | redact | Grievance/case numbers and tracking identifiers. |
| Signatures | redact | Handwritten or imaged signatures. |
| Account, session, and device data | redact | Account IDs, session tokens, cookies, device IDs, credentials. |
| Metadata | review | File metadata, authorship, timestamps, and provenance; redact when identifying. |

Categories marked `review` still default to redaction whenever the value could re-identify a person, case, or location. A reviewer may clear a `review` item only with a written, non-private justification (for example, "year only, not identifying").

## PASS / WARN / FAIL Handling

Each category, and the document overall, is classified per review:

- `PASS`: every required category was reviewed, all sensitive values are redacted or cleared with written justification, and GitHub-safe proof can be produced without exposing private details.
- `WARN`: a category has an ambiguity (unclear identifiability, missing reviewer justification, or unclear scope) that must be resolved in writing before any later content-eligible step.
- `FAIL`: a required category was not reviewed, sensitive values remain unredacted, detection was treated as approval, or proof would require private content.

Any `FAIL` is a **do not proceed** blocker. Any unresolved `WARN` is also a **do not proceed** blocker for content-eligible work. Stop before any real content is touched if any blocker is present.

## Reviewer Roles

- **Operator / approver.** Owns the approval packet and the final go/no-go. Confirms scope is one document and one gate. Signs and dates approval.
- **Redaction reviewer.** Performs the category-by-category human review, records PASS/WARN/FAIL per category, and writes non-private justifications for any cleared `review` item.
- **Escalation owner.** Resolves any `WARN` or disputed category and any sensitive edge case (medical, settlement, discipline, witness) before sign-off.

Roles may be the same person on a laptop-only pilot, but each role's decision must be recorded separately. Detection tooling (future) is never a reviewer and never grants approval.

## Escalation Rules

- Any `WARN` escalates to the escalation owner and must be resolved in writing before proceeding.
- Any medical, settlement, discipline, or witness content escalates for a second confirmation regardless of initial classification.
- Any uncertainty about identifiability defaults to redaction and escalates for review.
- Any request to clear a category in bulk, skip a category, or auto-approve detection is rejected and escalated.
- Any proof-safety concern (a field that could leak private content) escalates before proof is recorded.

## Default Deny and Not-Indexable Behavior

- The document is `not_indexable` until copy, redaction, metadata, and operator gates all pass in a later reviewed phase.
- Unreviewed or `WARN`/`FAIL` content remains blocked from any index, embedding, vector store, upload, export, or search availability.
- Eligibility never changes by implication; it requires explicit operator approval naming one gate and one document.

## Deletion and Retention Rules

- Retention defaults to deletion of any private working artifact (copies, detection output, review notes) once the review is complete or abandoned.
- If retention is approved, the future approval packet must name a private retention period, an owner, and a review date.
- Private artifacts are never committed, exported, uploaded, or indexed by default.
- Rollback for any gate can delete private working artifacts before any retry, and a retry never reuses unsafe artifacts.
- The real source is never mutated.

## GitHub-Safe Proof Rules

Allowed proof fields:

- Phase name.
- Commit SHA.
- Command names.
- PASS/WARN/FAIL states.
- Category names and outcome labels.
- Counts by safe category.
- Opaque pilot ID.
- Booleans such as `real_document_access=false` for planning phases.

Forbidden proof fields:

- Raw document text.
- Snippets.
- Private paths.
- Filenames when identifying.
- Member, employee, witness, officer, case, account, session, device, contact, or credential identifiers.
- OCR text.
- Hash values when sensitive.
- Exports.
- Backups.
- Private databases.
- Vector data.
- Embeddings.
- Private notes.
- Screenshots of real content.
- Any field that lets a reviewer reconstruct or locate the private source.
