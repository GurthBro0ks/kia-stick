# KIA Stick v0.6.0 Real-Doc Pilot Plan

Phase: `KIA-Stick-v0.6.0-real-doc-pilot-plan-only`

Status: `PLAN ONLY`

Baseline: `origin/main` verified at `ef1cb84`, the pushed v0.5.10 docs release pack.

Product version impact: none. `productVersion` stays `0.4.0`; any future product milestone bump must be approved in a separate implementation phase.

Model recommendation: Claude for the privacy and safety plan. GPT/Codex is acceptable for docs and tests. GLM-5.2 is optional for second-opinion review only.

## Plan-Only Boundary

This document is a design and approval checklist only. It does not authorize or implement real-document reads, directory listing, file selection, path reading, copying, OCR, text extraction, summarization, transformation, embeddings, indexing, vector storage, uploads, private-vault inspection, exports, service changes, pushes, or private-data commits.

No real document path, filename, text, snippet, image, OCR output, identifier, account data, session data, hash, export, vector data, or private note is needed to complete this phase.

## Future Pilot Shape

The safest future pilot is a single-document workflow with explicit operator confirmation before each irreversible or content-touching step. It should prove the gates, audit trail, rollback path, deletion path, and GitHub-safe proof model before any batch, folder, recursive, OCR, indexing, or automation feature is considered.

The future pilot must use an opaque pilot ID, for example `pilot-001`, instead of a source path or document name in logs intended for GitHub-safe review.

## Allowed Future Source Scope

A later implementation prompt may approve exactly one source item only if all of these are true:

- The operator explicitly selects one local file for a one-time pilot.
- The operator confirms they are allowed to review that file on this laptop.
- The file is relevant to the KIA Stick research domain.
- The prompt states the maximum allowed actions before any content is touched.
- The prompt states whether copying, hashing, text extraction, OCR, redaction review, metadata review, or indexing is authorized.
- The prompt forbids batch selection, folder selection, recursive scans, automatic discovery, cloud sync, upload handling, service changes, and private-data commits.

Nothing in this plan authorizes the future source action by itself.

## Stop Conditions

Stop before any real content is touched if any condition is true:

- The operator did not give a fresh written approval for this exact phase.
- The target phase does not include `real-doc-pilot` and `single-document`.
- The prompt asks for a folder, directory, batch, mount, archive, export set, recursive scan, or automatic discovery.
- The prompt asks to open, OCR, summarize, transform, embed, index, upload, or publish content before quarantine and redaction gates are complete.
- The prompt asks to commit real content, raw text, screenshots, exports, vector data, private notes, identifiers, account data, session data, secrets, or private paths.
- The file appears to contain medical, discipline, settlement, witness, grievance, account, session, device, credential, or other high-risk material and no explicit handling rule exists.
- The operator cannot explain the rollback and deletion path before starting.
- The repo has dirty unrelated changes, untracked private data, or tracked private layout files.
- Any proof artifact would need a raw path, filename, document text, OCR text, snippet, identifier, or sensitive hash to explain the result.

## Operator Approval Gates

Each future gate must require a visible operator approval and must record only GitHub-safe status facts.

| Gate | Approval question | Allowed future action after approval | GitHub-safe proof |
| --- | --- | --- | --- |
| 0. Phase approval | Is this exact single-document pilot approved? | Start the pilot harness only. | Phase name, operator decision status, no path. |
| 1. Source scope | Is exactly one operator-selected source item in scope? | Store an opaque pilot ID and source category label. | `pilot_id`, source category, no filename or path. |
| 2. Quarantine copy | Is a private quarantine copy approved? | Copy one file into ignored private storage only. | Copy gate status, private storage label, no path. |
| 3. Hash and provenance | Is private hash/provenance capture approved? | Compute hashes and provenance in private metadata only. | Hash-present boolean; omit hash if sensitive. |
| 4. Redaction detection | Is detection allowed for this document type? | Run approved local detection only. | Finding counts by category and severity only. |
| 5. Redaction review | Has a reviewer approved the redacted copy? | Mark redaction review outcome. | Reviewer role label and outcome only. |
| 6. Metadata review | Is authority/sensitivity metadata approved? | Mark metadata completeness and source category. | Metadata status labels only. |
| 7. Index eligibility | Is this redacted copy eligible for index? | Create index candidate metadata only if allowed. | Eligible/not eligible plus reason code only. |
| 8. Audit/proof | Is GitHub-safe proof reviewed? | Write sanitized proof summary. | Counts, booleans, result, and command names only. |
| 9. Retention/deletion | Is retention or deletion approved? | Keep or delete private pilot artifacts per decision. | Retention decision label only. |

## Future Quarantine Copy Rules

- Copy only after a written gate says copying is allowed.
- Copy one selected file only; never a folder, batch, archive, recursive tree, or auto-discovered item.
- Write the copy only under ignored private storage outside tracked Git content.
- Never mutate the source file.
- Never stage private quarantine storage, redacted outputs, OCR outputs, exports, backups, vector data, or private metadata.
- Never print the source path or quarantine path in terminal output intended for the proof pack.
- If a command fails after a copy, stop and run rollback before retrying.

## Hashing And Provenance

- Compute raw-file hash and quarantine-copy hash only if that gate is approved.
- Treat hashes as potentially sensitive when they could identify a known document.
- Store hashes in private pilot metadata by default.
- GitHub-safe proof may say `hashes_captured=true`; it must not include hash values unless the operator explicitly classifies those hashes as safe.
- Provenance must use source category labels and gate timestamps, not raw paths, filenames, or identifying notes.

## Redaction Review

The future pilot must separate detection from approval:

- Detection findings are suggestions, not redaction approval.
- Redaction review must be a human gate.
- Redacted output is not indexable until metadata review and index eligibility are also approved.
- Review notes must stay private if they include document facts, names, case details, locations, dates, or strategy.
- GitHub-safe proof may include counts by redaction category and final outcome labels only.

## Metadata Review

Metadata review must confirm:

- Source category.
- Authority level.
- Sensitivity level.
- Redaction status.
- Reviewer role.
- Index eligibility candidate status.
- Retention decision.
- Audit reference.

Metadata must not include raw document paths, filenames, quoted text, identifying names, case IDs, member identifiers, account/session data, or private notes in GitHub-safe outputs.

## Index Eligibility

The default eligibility state is `not_indexable`.

Index eligibility may change only after these proofs exist privately:

- Quarantine copy gate approved.
- Redaction review complete.
- Metadata review complete.
- Sensitive fields resolved.
- Operator approves index eligibility for this exact pilot item.

Even then, the first pilot should stop at an index candidate record unless a later written prompt explicitly authorizes real indexing. No vector store is approved by this plan.

## Audit, Rollback, And Retention

Future audit logs must be append-only for gate decisions, but GitHub-safe summaries must contain only labels, counts, booleans, command names, timestamps, and PASS/WARN/FAIL states.

Rollback must be available after every gate:

- Before copy: clear in-memory pilot state.
- After copy: delete the private quarantine copy and private metadata.
- After hash/provenance: delete private hash metadata.
- After redaction review: delete redacted working copy and private review notes.
- After metadata review: delete private metadata record.
- After index candidate: delete candidate metadata and confirm no vector data exists.
- After proof: remove any unsafe proof artifact and regenerate a sanitized summary.

Retention must default to deletion unless the operator explicitly approves a private retention period. GitHub-safe docs may record only the retention decision label, not the retained location.

## GitHub-Safe Proof Rules

Allowed in proof:

- Phase name.
- Commit SHA.
- Command names and PASS/WARN/FAIL.
- Counts and status labels.
- Opaque pilot ID.
- Gate names and outcome labels.
- Booleans such as `real_document_access=false` for this planning phase.

Forbidden in proof:

- Raw document text.
- Raw source paths or quarantine paths.
- Filenames if identifying.
- Snippets.
- OCR text.
- Screenshots of real content.
- Member, employee, witness, officer, case, account, session, device, or contact identifiers.
- Hash values when sensitive.
- Exports, backups, private databases, vector-store data, embeddings, or private notes.
- Any proof artifact that lets someone reconstruct the document or locate the private source.
