# KIA Stick v0.3 Build Identity Version System

- Phase: `KIA-Stick-v0.3-build-identity-version-system`
- Target machine: `USER_LAPTOP_ONLY`
- Scope: app version identity only; fake-doc corpus/index/prompt versions stay separate.
- Real-document access: none.

## Version Rule

KIA Stick now separates slow product milestone semver from per-build identity.

- `productVersion`: feature milestone version. For this phase it remains `0.3.0`.
- `channel`: release lane, default `dev`.
- `buildDate`: UTC `YYYYMMDD`, generated at runtime/build time.
- `gitSha`: short Git commit identity from `git rev-parse --short HEAD` when available, otherwise `unknown`.
- `displayVersion`: user-visible build identity in the form `productVersion-channel.buildDate+gitSha`.

Example:

```text
0.3.0-dev.20260620+c33c049
```

Small commits should change the Git SHA and therefore `displayVersion`; they should not bump `productVersion` unless a planned milestone phase changes.

## Exposed Fields

The version payload exposes:

- `productVersion`
- `channel`
- `buildDate`
- `gitSha`
- `displayVersion`
- `corpusVersion`
- `indexVersion`
- `promptVersion`
- `provider`

`corpusVersion`, `indexVersion`, and `promptVersion` are intentionally independent from the app build identity. A UI or source-corpus change can have a new `displayVersion` without changing corpus/index versions.

## Surfaces

- `/health` returns every version field at top level and inside the nested `version` object.
- `/version` displays `displayVersion` prominently and lists the full metadata payload.
- The app header, settings screen, answer footer, and saved answer metadata use `displayVersion`.
- Saved answers retain `displayVersion`, `corpusVersion`, `indexVersion`, `promptVersion`, and `provider` through the stored answer `version` object.

## Boundaries

This phase does not approve real-document reads, copies, scans, OCR, ingestion, summarization, transforms, indexing, uploads, or commits. `/media/mint/SHARED/APWU` and `~/kia-stick-private-vault` remain out of scope.
