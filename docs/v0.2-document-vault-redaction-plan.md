# KIA Stick v0.2 Document Vault and Redaction Plan

## Status

- Phase: `KIA-Stick-v0.2-document-vault-redaction-plan`
- Target machine: `USER_LAPTOP_ONLY`
- Target repo: `~/kia-stick`
- Status: planning only
- Baseline: v0.1 accepted at implementation commit `1d9b05c`, closeout commit `5a52c4e`, containment commit `29087a9`
- Current boundary: laptop-only, fake-doc-safe, no cloud/API keys, no NUCs
- Private vault boundary: real-document storage stays outside the repo, currently planned around `~/kia-stick-private-vault`

This document defines the future real-document governance model. It does not approve, implement, read, copy, scan, OCR, ingest, summarize, transform, index, or commit real APWU, USPS, member, local, case, or steward documents.

## Hard Boundaries

- Do not access `/media/mint/SHARED/APWU`.
- Do not inspect private vault contents during this planning phase.
- Do not recursively list, scan, copy, transform, OCR, index, summarize, or open real documents.
- Do not commit raw documents, screenshots, OCR dumps, captures, uploads, vector stores, exports, backups, tokens, cookies, account data, device data, API keys, or secrets.
- Do not add cloud services, Discord, SSH, NUC dependencies, Caddy, DNS, cron, systemd, tmux, API calls, or external sync.
- Future real-document work must be explicit, operator-selected, review-gated, and private by default.

## Document Lanes

### `official_public`

Official documents already public by source authority.

- Examples: public national agreements, public handbooks, public official notices, public LMOU material only when confirmed public.
- Default handling: eligible for metadata review and indexing after source authority is confirmed.
- GitHub handling: source text is not automatically GitHub-safe; only citations, metadata, summaries, and proof of review may be committed unless the document is confirmed redistributable.

### `official_member_only`

Official documents available to members but not intended as public web artifacts.

- Examples: member portal material, member-only national or local notices, private union reference material.
- Default handling: private vault only; index only after admin review confirms member-only use is acceptable on the local laptop.
- GitHub handling: never commit raw content, screenshots, exports, or OCR text.

### `local_official`

Official local documents controlled by the local or steward structure.

- Examples: local instructions, local minutes, local bulletins, local policy notes, local steward packets.
- Default handling: metadata review must identify authority, owner, date status, supersession risk, and distribution limit.
- GitHub handling: sanitized metadata only unless separately confirmed public and redistributable.

### `steward_only`

Steward work product, notes, issue analysis, annotated reminders, and private operating knowledge.

- Examples: issue notes, argument outlines, timelines, preparation checklists, non-public working notes.
- Default handling: private by default; reviewed steward notes may be index eligible only after sensitive fields are removed and authority is marked as non-controlling.
- GitHub handling: only clean-room templates, generic workflows, and sanitized metadata may be committed.

### `redacted_examples`

Approved examples derived from real patterns after redaction and review.

- Examples: anonymized issue patterns, redacted case-style examples, generic evidence checklists.
- Default handling: index eligible only after redaction review, metadata review, and admin approval.
- GitHub handling: may be committed only when the example contains no real names, identifiers, screenshots, source text, private facts, or re-identifying combinations.

### `restricted_sensitive`

Documents or extracts with heightened privacy, legal, medical, employment, or case sensitivity.

- Examples: medical material, discipline files, settlements, witness statements, personnel records, screenshots, grievances with identifiable facts, private emails, officer or management names tied to specific cases.
- Default handling: never index by default; store only in private vault lanes if a future phase explicitly allows it.
- GitHub handling: never commit raw content, redacted copies pending review, OCR, screenshots, exports, or detailed metadata that can identify the person or case.

### `quarantine`

Temporary private holding area for raw operator-selected copies before review.

- Examples: future copied-in files awaiting hashing, provenance, redaction detection, review, and disposition.
- Default handling: no indexing, no OCR, no summarization, no model access, no export, no Git staging.
- GitHub handling: never commit quarantine files or generated content from quarantine.

## Future Lifecycle

The future real-document flow must be implemented as a gated workflow:

```text
operator-selected path
-> explicit future copy to quarantine
-> hash/provenance record
-> redaction detection
-> admin review
-> approved redacted copy
-> authority metadata
-> index eligibility decision
-> audit log
```

Required gates:

- Operator selection must be explicit. No recursive folder pickers, background scans, or automatic imports.
- Copying to quarantine must be a separate confirmed action. Selecting a path does not copy content.
- Quarantine files must be immutable for review. Any redacted derivative must be a new file with its own hash.
- Provenance must be captured before redaction: source lane, selected filename, private source note, copy timestamp, operator, raw hash, and quarantine hash.
- Redaction detection must flag likely sensitive fields before a human review. Detection is advisory; it does not approve anything.
- Admin review must decide lane, sensitivity, redaction status, authority level, supersession risk, and whether the item can become an approved redacted copy.
- Approved redacted copies must be stored separately from quarantine.
- Authority metadata must distinguish controlling, persuasive, local official, steward note, example, and unknown authority.
- Index eligibility must be an explicit yes/no decision, not an implied result of approval.
- Audit logging must record each state transition without writing raw content into GitHub-safe logs.

## Redaction Fields

Future redaction detection and review must check at least these fields:

- Member names.
- Addresses.
- Phone numbers and email addresses.
- Employee IDs and other work identifiers.
- Medical details, restrictions, notes, diagnoses, accommodations, and leave details.
- Discipline facts, charges, penalties, suspensions, removals, and investigative details.
- Settlement terms, amounts, conditions, non-disclosure language, and case outcomes.
- Witness statements, witness names, witness contact data, and identifying context.
- Screenshots, chat logs, email headers, embedded images, filenames, EXIF metadata, and visible account details.
- Management names and officer names when tied to a specific matter.
- Dates that can re-identify a case, event, meeting, charge, or settlement.
- Installation, station, branch, unit, route, bid, finance number, or local workplace identifiers.
- Case IDs, grievance numbers, arbitration numbers, tracking numbers, docket numbers, and file numbers.

Review should also flag combinations that are identifying even when each field looks harmless alone.

## Metadata Model

Minimum future metadata fields:

- `document_id`: stable local ID, not a source filename.
- `lane`: one of the approved lane names.
- `authority_level`: controlling, official, local_official, persuasive, steward_note, example, or unknown.
- `source_status`: public, member_only, local_private, steward_private, restricted, or unknown.
- `sensitivity`: low, moderate, high, restricted_sensitive, or unknown.
- `raw_hash`: private hash for quarantine material.
- `redacted_hash`: hash for approved redacted derivative when present.
- `provenance_note`: private operator-entered context, not committed.
- `redaction_status`: not_started, detection_flagged, reviewed_needed, approved_redacted, rejected, or restricted.
- `metadata_status`: not_reviewed, needs_changes, reviewed, or rejected.
- `index_eligibility`: not_eligible, eligible_private, eligible_redacted, or eligible_public.
- `reviewer`: local admin identity or initials.
- `reviewed_at`: local timestamp.
- `audit_log_id`: local audit reference.

No metadata field may store tokens, account identifiers, member identifiers, raw path dumps, or private narrative content in GitHub-safe files.

## Index Rules

Indexing must be opt-in after review.

Allowed after review:

- Reviewed official docs with confirmed authority and acceptable source status.
- Approved redacted examples with no re-identifying fields.
- Reviewed steward notes that are clearly marked non-controlling and sanitized.

Never index:

- Quarantine files.
- Raw uploads.
- Unreviewed sensitive documents.
- Medical documents.
- Screenshots.
- OCR dumps.
- Private exports.
- Raw member-only material before admin review.
- Settlement files before explicit restricted review.
- Witness statements or investigative material.
- Anything with unresolved redaction flags.

Index output must inherit the strictest source lane and sensitivity label from the source material.

## Ignored Private Layout

The repo must continue to ignore these paths:

- `DB/`
- `data/real-documents/`
- `data/quarantine/`
- `data/redacted-approved/`
- `exports/`
- `backups/`
- `vector-store/`

Future tooling may create private local directories with these names, but those paths must remain untracked and excluded from GitHub-safe proof packs.

## Future UI Surfaces

Future UI work should separate governance states into dedicated surfaces:

- Vault: private document inventory and lane overview.
- Quarantine: raw copied-in files awaiting hash, provenance, and review.
- Redaction Review: detection flags, redaction decisions, and approved derivative creation.
- Metadata Review: authority, source status, sensitivity, supersession, and reviewer fields.
- Index Eligibility: explicit index yes/no queue with reason codes.
- Audit Log: immutable local event history for selection, copy, hash, review, approval, rejection, and indexing decisions.

UI copy must avoid implying that upload, quarantine, redaction, or review automatically makes content indexable.

## Future Proof Rules

GitHub-safe proof packs may include:

- Command logs.
- Git status output.
- Test, lint, typecheck, build, and QA output.
- Counts of reviewed items by lane.
- Hashes of approved redacted derivatives only when the hash itself does not reveal source path or private details.
- Sanitized metadata schemas and validation summaries.

GitHub-safe proof packs must not include:

- Raw documents.
- Raw screenshots.
- OCR text.
- Private source paths.
- Member names or identifiers.
- Case identifiers.
- Medical, discipline, settlement, witness, or grievance details.
- Vector-store files.
- Private exports or backups.
- Logs that include real document content.

## GitHub-Safe Rules

Allowed to commit:

- Planning docs.
- Source code and tests.
- Fake sample documents.
- Sanitized metadata schemas.
- Sanitized reports.
- Proof summaries that contain no raw real-document content.

Not allowed to commit:

- Real APWU, USPS, member, local, case, steward, or management documents.
- Quarantine content.
- Approved redacted files before a separate public-safety review.
- OCR dumps, embeddings, vector stores, exports, or backups.
- Private vault databases.
- Secrets, tokens, API keys, auth headers, cookies, account data, or device data.
