# KIA Stick v0.8.4 Large Bundle Operator Workflow

Phase: `KIA-Stick-v0.8.4-large-bundle-operator-workflow`

Status: operator workflow documentation

Product version: `0.7.0`

Prompt version: `prompt.fake-docs.v0.5-import-wizard-hardening`

## Scope

This phase documents the larger-chunk workflow used by the v0.8.1 through v0.8.5 bundle. It replaces one micro-phase QA loop at a time with one proof-driven bundle review when the work is fake/synthetic/docs/tooling only.

## Large-Bundle Rule

An agent may complete multiple fake/synthetic subphases before operator QA when all of these are true:

- Every subphase is fake/synthetic/docs/tooling/state only.
- Product version and prompt version do not change unless a separate approved product-version phase exists.
- The work adds no runtime real-doc capability.
- `queue-015-v07-first-real-doc-gate-request` stays blocked.
- The proof directory captures before/after queue state, focused tests, full validation, safety-boundary scans, and git state.
- The agent stops on any WARN, FAIL, safety-boundary issue, unrelated dirty-state conflict, or validation blocker.

## Operator QA Flow

The bundle gets one operator QA review after the subphases are complete and validation passes.

The operator review should check:

- `RESULT.md`
- `SUBPHASES.md`
- `queue:list` before and after
- `queue:next` before and after
- focused test logs
- full validation logs
- fake/privacy scan logs
- safety-boundary scan logs
- changed files
- git status after
- copy-ready OPERATOR_QA_PASS text

One `OPERATOR_QA_PASS` applies to the whole bundle only if every subphase stayed inside the fake/synthetic boundary.

## Closeout And Push

After one bundle-level operator QA PASS, closeout/push can happen as a separate follow-up. This v0.8.4 phase does not push by itself.

The closeout follow-up should make one task-scoped commit if validation passes, then push only when the operator explicitly authorizes it.

## Real-Doc Work Remains Separate

Real-doc work is not part of the large fake/synthetic bundle rule.

Any future real-doc packet still requires:

- exactly one document
- exactly one gate
- a separate approval packet
- fresh operator approval
- PASS safety checklist result
- stop-on-WARN/FAIL proof
- GitHub-safe proof exclusions
- no recursive source scope

## Clear Answer

Fake/synthetic subphases can be batched into one proof pack and one operator QA pass; real-doc work cannot be batched into that workflow.

- One bundle QA PASS is not a push authorization by itself.
