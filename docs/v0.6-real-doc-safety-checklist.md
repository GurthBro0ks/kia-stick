# KIA Stick v0.6.2 Real-Doc Safety Review Checklist

Phase: `KIA-Stick-v0.6.2-safety-review-checklist`

Status: `PLAN ONLY`

Baseline: `origin/main` verified at `7b2d5b4`, the pushed v0.6.1 post-plan safety closeout.

Product version impact: none. `productVersion` stays `0.4.0`; any future product milestone bump must be approved in a separate implementation phase.

Model recommendation: Claude for checklist and safety review. GPT/Codex is acceptable for docs and tests. GLM-5.2 is optional for second-opinion review only.

## Plan-Only Boundary

This checklist is a review artifact only. It does not authorize or implement real-document reads, directory listing, file selection, path reading, copying, OCR, text extraction, summarization, transformation, embeddings, indexing, vector storage, upload handling, private-vault inspection, exports, service changes, pushes, or private-data commits.

No real document path, filename, text, snippet, image, OCR output, identifier, account data, session data, hash, export, vector data, or private note is needed to complete this phase.

The only approved output of this phase is GitHub-safe docs, tests, queue state, and closeout state.

## Review Rule

Before any future real content is touched, the operator must classify every gate below as `PASS`, `WARN`, or `FAIL`.

- `PASS`: the gate is explicit, narrow, reversible, and GitHub-safe proof can be produced without exposing private details.
- `WARN`: the gate has an ambiguity that must be resolved in writing before the next content-touching action.
- `FAIL`: the gate is missing, too broad, not reversible, or would require private content in proof.

Any `FAIL` is a **do not proceed** blocker. Any unresolved `WARN` is also a **do not proceed** blocker for content-touching work.

## Pre-Content Blockers

Stop before any real content is touched if any item is true:

- The phase name, target repo, target machine, or allowed action set is missing.
- Operator approval is stale, implied, pasted from an older run, or not tied to the exact phase.
- The source scope includes a folder, batch, archive, export set, recursive rule, auto-discovery, synced location, cloud source, or private repository.
- The requested action includes file picker, path reader, file open, copy, OCR, extraction, summarization, transformation, embeddings, indexing, upload handling, vector store, export, service change, push, or private-data commit before the relevant gate is approved.
- The future prompt cannot explain rollback and deletion before starting.
- The repo has unrelated dirty changes, untracked private artifacts, or tracked private storage paths.
- Proof would need raw text, snippets, private paths, identifiers, OCR text, sensitive hashes, exports, vector data, or private notes to justify the result.

## Gate Checklist

| Gate | PASS | WARN | FAIL |
| --- | --- | --- | --- |
| Operator approval | Fresh written approval names the exact phase, one-document pilot intent, allowed actions, blocked actions, and rollback expectation. | Approval exists but has ambiguous wording or omits one blocked-action family. | Approval is missing, generic, inherited from a prior phase, or allows broad real-doc work. |
| Source scope | Exactly one operator-selected local source item is in scope, represented by an opaque pilot ID in GitHub-safe outputs. | One source is intended, but source category, authority, or permission is not written clearly. | Any folder, batch, archive, recursive scan, automatic discovery, private repo, mount scan, cloud source, or upload source is requested. |
| Single-document limit | The prompt states one document only and blocks batch expansion until a later reviewed phase. | The prompt says one document but leaves attachments, companion files, or derived exports ambiguous. | More than one item can be selected, discovered, copied, OCRed, extracted, summarized, indexed, or exported. |
| Non-recursive rule | The prompt explicitly forbids recursive traversal, directory enumeration, and auto-discovery. | Non-recursive intent is implied but not written as a hard gate. | Directory listing, recursive traversal, tree scan, folder picker, globbing, archive expansion, or auto-discovery is allowed. |
| Quarantine destination | Any future copy destination is ignored private storage, named only by a safe label in proof, and not inside tracked Git content. | Destination is private but ignore status, retention owner, or proof label is unclear. | Destination is tracked, public, printed as a private path, or mixed with exports, backups, vector data, or repo docs. |
| Hash and provenance | Hash/provenance capture is separately approved, stored privately by default, and GitHub proof uses booleans or safe labels. | Hashes are useful but sensitivity classification or storage location is unclear. | Hash values, filenames, raw paths, identifying provenance, or source notes are required in public proof. |
| Redaction review | Detection is separate from human review, review notes stay private, and only counts/outcome labels can enter proof. | Reviewer role or finding categories are unclear but no content is exposed. | Redaction is treated as automatic approval, or proof requires snippets, OCR text, names, case facts, or private notes. |
| Metadata review | Authority, sensitivity, redaction status, reviewer role, retention decision, and audit reference are reviewed as labels. | Metadata labels exist but one label family lacks a reviewer decision. | Metadata includes private paths, filenames, quoted text, identifiers, account/session data, or document facts. |
| Index eligibility | Default is `not_indexable`; eligibility can change only after copy, redaction, metadata, and operator gates all pass. | Eligibility is still candidate-only and needs operator wording before any later index step. | Real indexing, embeddings, vector store, upload, or search availability is approved by implication. |
| Audit | Audit uses append-only gate decisions with labels, counts, booleans, command names, timestamps, and PASS/WARN/FAIL states only. | Audit has safe fields but lacks one required gate outcome. | Audit stores or prints raw paths, text, snippets, OCR, identifiers, sensitive hashes, exports, vector data, or private notes. |
| Rollback | Rollback is written for every gate and can delete private working artifacts before retrying. | Rollback exists but does not assign an operator decision for one artifact class. | No rollback exists, source mutation is possible, or retry would reuse unsafe artifacts. |
| Deletion and retention | Retention defaults to deletion unless the operator approves a private retention period and owner. | Retention label exists but review date or owner is missing. | Private artifacts are retained by default, committed, exported, uploaded, indexed, or left without deletion instructions. |
| Stop conditions | Stop conditions are listed before content-touching commands and include proof-safety blockers. | Stop conditions exist but do not name every blocked action family. | Stop conditions appear after content-touching steps or omit private proof blockers. |

## GitHub-Safe Proof Checklist

Allowed proof fields:

- Phase name.
- Commit SHA.
- Command names.
- PASS/WARN/FAIL states.
- Gate names and outcome labels.
- Counts by safe category.
- Opaque pilot ID.
- Booleans such as `real_document_access=false` for planning phases.

Forbidden proof fields:

- Raw document text.
- Snippets.
- Private paths.
- Filenames when identifying.
- Member, employee, witness, officer, case, account, session, device, contact, or credential identifiers.
- OCR text.
- Hash values when sensitive.
- Exports.
- Backups.
- Private databases.
- Vector data.
- Embeddings.
- Private notes.
- Screenshots of real content.
- Any field that lets a reviewer reconstruct or locate the private source.

## Do-Not-Proceed Decision Matrix

| Result | Meaning | Required action |
| --- | --- | --- |
| `PASS` all gates | Future prompt may proceed only to the next separately approved gate. | Record the safe proof summary and wait for operator approval before content-touching work. |
| Any `WARN` | Ambiguity exists. | Resolve the ambiguity in docs or operator approval, then rerun the checklist. |
| Any `FAIL` | Safety blocker exists. | Stop. Do not touch real content. Fix scope, rollback, proof, or approval before retrying. |
| Proof unsafe | Public evidence would leak private details. | Delete/regenerate proof using labels, counts, booleans, and safe command names only. |
| Rollback unclear | Private artifacts may remain unmanaged. | Write deletion and retention instructions before approving any content-touching action. |
