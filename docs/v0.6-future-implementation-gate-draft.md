## Purpose

This document drafts the **future implementation gate** structure for any later one-document real-doc pilot. It defines how a future prompt must request exactly one gate and exactly one document, what fields that prompt must contain, what gate types exist, and what blocks the request from proceeding.

This draft is non-executable. It does not implement, authorize, or start any real-document handling. It only specifies the shape a future, separately approved prompt must take before any content-touching work can be considered.

## Plan-Only Boundary

This gate draft is a planning artifact only. It does not authorize or implement real-document reads, directory listing, file selection, file pickers, path reading, copying, OCR, text extraction, summarization, transformation, detection over real content, real redaction, embeddings, indexing, vector storage, upload handling, private-vault inspection, exports, service changes, pushes, or private-data commits.

No real document path, filename, text, snippet, image, OCR output, identifier, account data, session data, hash, export, vector data, or private note is needed to complete this phase, and none may be pasted into a completed future prompt or its proof.

The only approved output of this phase is GitHub-safe docs, tests, queue state, and closeout state.

## One-Gate / One-Document Rule

A future implementation prompt may request **exactly one gate** for **exactly one document** per run.

- Exactly one gate type (from the Gate Types list below) may be requested.
- Exactly one operator-selected document, referenced by an opaque pilot ID, may be in scope.
- No batch, folder, archive, recursive scan, auto-discovery, companion file, or derived export may be included.
- Advancing to the next gate requires a new, separately approved prompt.

Any prompt that bundles multiple gates, multiple documents, or open-ended scope is rejected and must be split.

## Required Future Prompt Fields

A future implementation prompt must specify all of the following. Any missing field is a `do not proceed` blocker.

| Field | Requirement |
| --- | --- |
| Exact gate | Names exactly one gate type from the Gate Types list. |
| Exact one-document scope | Names exactly one operator-selected document by opaque pilot ID; blocks batch, recursion, and companions. |
| Allowed action | States the single narrow, reversible action approved for that one gate. |
| Blocked actions | Explicitly blocks every action family not named in the allowed action. |
| Approval packet reference | References a completed, signed packet from `docs/v0.6-operator-approval-packet.md`. |
| Safety checklist result | Records a `PASS` result for `docs/v0.6-real-doc-safety-checklist.md` against this scope. |
| Redaction policy result | Records redaction handling per `docs/v0.6-local-redaction-policy-plan.md` (PASS/WARN/FAIL by category). |
| Rollback | Written rollback for the named gate that can delete private working artifacts before retry; source never mutated. |
| Deletion and retention | Default deletion of private working artifacts; any retention names period, owner, and review date. |
| Proof-safe output | Proof uses only labels, counts, booleans, gate names, and PASS/WARN/FAIL states; no private content. |
| Stop conditions | Lists do-not-proceed blockers before any content-touching command. |

## Gate Types

A future prompt may request exactly one of these gate types per run. They are listed in their natural review order, but each requires its own separate approval:

1. **Source selection.** Operator selects one document; it is referenced only by an opaque pilot ID. No listing, traversal, or picker over a directory.
2. **Quarantine copy.** A single approved copy is placed in ignored private storage, named only by a safe label in proof; never tracked in Git.
3. **Provenance and hash.** Provenance/hash capture is separately approved, stored privately by default; proof uses booleans or safe labels only.
4. **Redaction detection.** Automated detection proposes candidates only; detection is never approval.
5. **Redaction review.** A human reviewer confirms every redaction outcome by category; review notes stay private; proof uses counts and outcome labels only.
6. **Metadata review.** Authority, sensitivity, redaction status, reviewer role, retention decision, and audit reference are reviewed as labels only.
7. **Index eligibility.** Default is `not_indexable`; eligibility can change only after copy, redaction, metadata, and operator gates all pass.
8. **Audit.** Append-only gate decisions recorded with labels, counts, booleans, command names, timestamps, and PASS/WARN/FAIL states only.
9. **Rollback.** Deletes private working artifacts for the named gate before any retry; the real source is never mutated.
10. **Deletion.** Retention defaults to deletion unless an approved private retention period, owner, and review date are named.

## Stop Conditions / Do-Not-Proceed Blockers

Stop before any real content is touched if any item is true:

- The prompt requests more than one gate, more than one document, or open-ended scope.
- Any required future-prompt field above is blank, generic, inherited, or undated.
- The approval packet is missing, unsigned, undated, stale, or not tied to the exact gate and document.
- The safety checklist result is not `PASS` for this scope.
- The redaction policy result is missing, `WARN` (unresolved), or `FAIL`.
- The requested action includes file picker, path reader, file open, copy, OCR, extraction, summarization, transformation, embeddings, indexing, upload handling, vector store, export, service change, push, or private-data commit beyond the one named gate's allowed action.
- Rollback, deletion, or retention is unwritten.
- The repo has unrelated dirty changes, untracked private artifacts, or tracked private storage paths.
- Proof would need raw text, snippets, private paths, identifiers, OCR text, sensitive hashes, exports, vector data, or private notes to justify the result.

Any blocker is a **do not proceed** condition. Stop, fix the gap, and re-request through a separately approved prompt.

## GitHub-Safe Proof Rules

Allowed proof fields:

- Phase name.
- Commit SHA.
- Command names.
- PASS/WARN/FAIL states.
- Gate names and outcome labels.
- Counts by safe category.
- Opaque pilot ID.
- Booleans such as `real_document_access=false` for planning phases.

Forbidden proof fields:

- Raw document text.
- Snippets.
- Private paths.
- Filenames when identifying.
- Member, employee, witness, officer, case, account, session, device, contact, or credential identifiers.
- OCR text.
- Hash values when sensitive.
- Exports.
- Backups.
- Private databases.
- Vector data.
- Embeddings.
- Private notes.
- Screenshots of real content.
- Any field that lets a reviewer reconstruct or locate the private source.

## What This Draft Does Not Authorize

This gate draft does not approve real detection, real redaction, real import, file pickers, path readers, file reads, copying, OCR, text extraction, summarization, transforms, embeddings, indexing, vector stores, upload handlers, private-vault inspection, cloud calls, service changes, pushes, or private document access.

- That prompt must define its own rollback, deletion/retention, and GitHub-safe proof before any content-touching command.
